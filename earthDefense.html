Earth Defense

Java based game written in processing

Purpose

This was a game written in processing, in combinations with Schulich Ignite to demonstrate what could be done using java and the processing library. Ignite offered free programming to high school and university students, and as a mentor and a lecturer, I wanted to demonstrate why someone should continue with programming despite its difficulties. One of the most impressive ways I could show it was by creating a game, where is used many aspects of what we taught just at a higher level.

Technical aspects

The first version of this game was actually written when I was a mentee of Schulich Ignite, there were aspects of coding which I didn't understand at the time and wanted to prove to myself to show how much of improvement as a programer and a game desginer.

First was a good way to store all the enemies and bullets, a linkedList was choosen to do this. A linkedList was chosen over an arrayList because when an element (enemies and bullets) was removed from it, all of the elements in an arrayList would have to be recreated in order to remove an element. This a costly operation and removing from the datastructure would happen often, thus a linkedList was choosen. Since a linkedList for insertion and deleting is in O(1) in worst cases, although searching is in O(n) but so is an arrayList. In hinde sight, I shoudl have come up with a better algorithm to deal with handling collision detection as each bullet had to iterate through the entire enemy linkedList. There must have been a been a better algorithm to reduce the number of times or how far it went into the enemies's linkedList.

During the creation I learned about inheritance, I wanted to implement some better programming features into my game. while not an easy or basic concept, I figured that since my first version used only an Array and iterated through it to do enemy logic, I should expand what it could do while still using this as the basic concept. Thus I made enemies and weapons abstract classes and then all enemy and weapon types inherited from this. This allowed me to easily create different and varied types of weapons and enemies. From these concepts I created 3 new enemies that were not in the original version, Brute, Gremlin and Swarm Host. These used the concepts I’d learned such as overloading functions and constructors to create different behaviours while still having the same “AI” driving it.

As I looked back on the code to write this section, I realized all the aspects that I could of done better to increase modularity and cohesion while decreasing coupling. Some things such as implementing interfaces to allow change how something behaviors rather than using the same logic for each type of enemy. There’s also some bad code due to a (at the time) lazy programmer, perhaps one day I’ll go back and rewrite the game a third time using the knowledge gained from more years of experience.
